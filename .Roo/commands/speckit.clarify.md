**说明：** 本命令用于检测当前功能需求中未明确的区域，通过提出最多 5 个高度针对性的澄清问题，并将答案编码回需求文件中。建议在执行 `/speckit.plan` 之前运行此命令。

## 用户输入

```text
$ARGUMENTS
```

在继续之前，**必须**考虑用户输入（如果非空）。

## 执行流程

目标：检测并减少活动功能需求中的歧义或缺失决策点，并将澄清直接记录在需求文件中。

注意：此澄清工作流预期在调用 `/speckit.plan` 之前运行（并完成）。如果用户明确表示跳过澄清（例如，探索性尖峰），你可以继续，但必须警告下游返工风险增加。

执行步骤：

1. **查找需求文档**：
   - 定位 PRD：`文档/开发文档/PRD/PRD-xxxxx-功能名称.md`
   - 如果 JSON 解析失败，中止并指示用户重新运行 `/speckit.specify` 或验证功能分支环境。

2. **加载当前需求文件**。使用此分类法执行结构化歧义和覆盖扫描。对于每个类别，标记状态：清晰 / 部分 / 缺失。生成用于优先级排序的内部覆盖图（除非不会提出问题，否则不输出原始图）。

   **功能范围与行为**：
   - 核心用户目标和成功标准
   - 明确的范围外声明
   - 用户角色/人物区分

   **领域与数据模型**：
   - 实体、属性、关系
   - 身份和唯一性规则
   - 生命周期/状态转换
   - 数据量/规模假设

   **交互与用户体验流程**：
   - 关键用户旅程/序列
   - 错误/空/加载状态
   - 无障碍或本地化说明

   **非功能质量属性**：
   - 性能（延迟、吞吐量目标）
   - 可扩展性（水平/垂直、限制）
   - 可靠性和可用性（正常运行时间、恢复期望）
   - 可观察性（日志、指标、追踪信号）
   - 安全和隐私（认证/授权、数据保护、威胁假设）
   - 合规/监管约束（如有）

   **集成与外部依赖**：
   - 外部服务/API 及故障模式
   - 数据导入/导出格式
   - 协议/版本假设

   **边界情况与故障处理**：
   - 负面场景
   - 速率限制/节流
   - 冲突解决（例如，并发编辑）

   **约束与权衡**：
   - 技术约束（语言、存储、托管）
   - 明确的权衡或拒绝的替代方案

   **术语与一致性**：
   - 规范术语表术语
   - 避免的同义词/弃用术语

   **完成信号**：
   - 验收标准可测试性
   - 可衡量的完成定义风格指标

   **杂项/占位符**：
   - TODO 标记/未解决的决策
   - 缺乏量化的歧义形容词（"健壮"、"直观"）

3. **生成（内部）优先级排序的候选澄清问题队列**（最多 5 个）。不要一次全部输出。应用这些约束：
    - 整个会话最多 10 个问题。
    - 每个问题必须可以用以下方式回答：
       - 简短的多选选择（2-5 个不同的、互斥的选项），或
       - 一个词/短语答案（明确约束："用 <=5 个词回答"）。
    - 仅包含答案实质性影响架构、数据建模、任务分解、测试设计、用户体验行为、运营就绪性或合规验证的问题。
    - 确保类别覆盖平衡：首先尝试覆盖影响最高的未解决类别。
    - 排除已回答的问题、琐碎的风格偏好或计划级执行细节（除非阻塞正确性）。
    - 优先考虑减少下游返工风险或防止验收测试不一致的澄清。
    - 如果超过 5 个类别未解决，按（影响 * 不确定性）启发式选择前 5 个。

4. **顺序提问循环（交互式）**：
    - 一次只展示**一个**问题。
    - 对于多选问题：
       - **分析所有选项**并根据以下因素确定**最合适的选项**：
          - 项目类型的最佳实践
          - 类似实现中的常见模式
          - 风险降低（安全、性能、可维护性）
          - 与规格中可见的任何明确项目目标或约束的一致性
       - 在顶部突出展示你的**推荐选项**，并附上清晰的理由（1-2 句话解释为什么这是最佳选择）。
       - 格式为：`**推荐:** 选项 [X] - <理由>`
       - 然后将所有选项渲染为 Markdown 表格：

       | 选项 | 描述 |
       |------|------|
       | A | <选项 A 描述> |
       | B | <选项 B 描述> |
       | C | <选项 C 描述>（根据需要添加 D/E，最多 5 个）|
       | 简短 | 提供不同的简短答案（<=5 个词）（仅在适合自由形式替代时包含）|

       - 表格后添加：`你可以用选项字母回复（例如，"A"），通过说"是"或"推荐"接受推荐，或提供你自己的简短答案。`
    - 用户回答后：
       - 如果用户回复"是"、"推荐"或"建议"，使用你之前陈述的推荐/建议作为答案。
       - 否则，验证答案映射到一个选项或符合 <=5 个词的约束。
       - 如果歧义，要求快速消歧（计数仍属于同一问题；不要前进）。
       - 一旦满意，将其记录在工作内存中（尚未写入磁盘）并移至下一个排队的问题。
    - 在以下情况下停止提问：
       - 所有关键歧义提前解决（剩余排队项变得不必要），或
       - 用户发出完成信号（"完成"、"好"、"不再"），或
       - 你达到 5 个已提问的问题。
    - 永远不要提前透露未来排队的问题。
    - 如果开始时不存在有效问题，立即报告无关键歧义。

5. **每个接受的答案后集成（增量更新方法）**：
    - 维护需求的内存表示（开始时加载一次）加上原始文件内容。
    - 对于此会话中的第一个集成答案：
       - 确保存在 `## 澄清` 章节（如果缺失，在规格模板中最高级别上下文/概述章节之后创建）。
       - 在其下创建（如果不存在）今天的 `### 会话 YYYY-MM-DD` 子标题。
    - 接受后立即追加一个项目符号行：`- 问: <问题> → 答: <最终答案>`。
    - 然后立即将澄清应用到最合适的章节：
       - 功能歧义 → 更新或在功能需求中添加项目符号。
       - 用户交互/角色区分 → 更新用户故事或角色子章节。
       - 数据形状/实体 → 更新数据模型。
       - 非功能约束 → 在非功能/质量属性章节添加/修改可衡量标准。
       - 边界情况/负面流程 → 在边界情况/错误处理下添加新项目符号。
       - 术语冲突 → 在规格中规范化术语。
    - 如果澄清使早期歧义声明无效，替换该声明而不是重复；不留下过时的矛盾文本。
    - 每次集成后保存需求文件以最小化上下文丢失风险（原子覆盖）。
    - 保留格式：不要重新排序不相关的章节；保持标题层次结构完整。
    - 保持每个插入的澄清最小且可测试（避免叙述漂移）。

6. **验证（每次写入后执行加最终通过）**：
   - 澄清会话每个接受的答案恰好包含一个项目符号（无重复）。
   - 总提问（接受）问题 ≤ 5。
   - 更新的章节不包含新答案本应解决的残留模糊占位符。
   - 没有矛盾的早期声明残留（扫描现在无效的替代选择已删除）。
   - Markdown 结构有效；仅允许的新标题：`## 澄清`、`### 会话 YYYY-MM-DD`。
   - 术语一致性：所有更新章节使用相同的规范术语。

7. **将更新的需求写回** PRD 文件。

8. **报告完成**（提问循环结束或提前终止后）：
   - 提问和回答的问题数量。
   - 更新的需求路径。
   - 触及的章节（列出名称）。
   - 覆盖摘要表，列出每个分类类别及状态：已解决（之前是部分/缺失且已处理）、延迟（超出问题配额或更适合规划阶段）、清晰（已经足够）、未解决（仍然部分/缺失但影响低）。
   - 如果任何未解决或延迟项残留，建议是继续 `/speckit.plan` 还是稍后在计划后再次运行 `/speckit.clarify`。
   - 建议的下一个命令。

行为规则：

- 如果未发现有意义的歧义（或所有潜在问题都是低影响的），回复："未检测到值得正式澄清的关键歧义。"并建议继续。
- 如果需求文件缺失，指示用户先运行 `/speckit.specify`（不要在此创建新需求）。
- 永远不要超过 5 个总提问的问题（单个问题的澄清重试不计为新问题）。
- 避免推测性技术栈问题，除非缺失阻塞功能清晰度。
- 尊重用户提前终止信号（"停止"、"完成"、"继续"）。
- 如果由于完全覆盖而未提问，输出紧凑的覆盖摘要（所有类别清晰）然后建议前进。
- 如果配额用尽但仍有未解决的高影响类别，在延迟下明确标记它们并附上理由。

优先级排序上下文: $ARGUMENTS
