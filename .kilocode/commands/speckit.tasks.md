**说明：** 本命令用于基于可用的设计文档（PRD、TDD、data-model.md 等）生成可执行的、依赖有序的任务列表文件 FD 功能文档。

## 用户输入

```text
$ARGUMENTS
```

在继续之前，**必须**考虑用户输入（如果非空）。

## 执行流程

1. **设置**: 
   - 查找对应的 PRD 和 TDD 文档
   - 确定文档编号
   - 创建 FD 文档路径：`文档/开发文档/FD/FD-xxxxx-功能名称.md`

2. **加载设计文档**: 从文档目录读取：
   - **必需**: TDD（技术栈、库、结构）、PRD（用户故事及优先级）
   - **可选**: data-model.md（实体）、contracts/（API 端点）、research.md（决策）、quickstart.md（测试场景）
   - 注意：并非所有项目都有所有文档。根据可用内容生成任务。

3. **执行任务生成工作流**:
   - 加载 TDD 并提取技术栈、库、项目结构
   - 加载 PRD 并提取用户故事及其优先级（P1、P2、P3 等）
   - 如果 data-model.md 存在：提取实体并映射到用户故事
   - 如果 contracts/ 存在：将端点映射到用户故事
   - 如果 research.md 存在：提取决策用于设置任务
   - 生成按用户故事组织的任务（见下方任务生成规则）
   - 生成显示用户故事完成顺序的依赖图
   - 为每个用户故事创建并行执行示例
   - 验证任务完整性（每个用户故事有所有需要的任务，可独立测试）

4. **生成 FD 功能文档**: 使用 `.specify/templates/tasks-template.md` 作为结构，填写：
   - 来自 TDD 的正确功能名称
   - 阶段 1: 设置任务（项目初始化）
   - 阶段 2: 基础架构任务（所有用户故事的阻塞性前置条件）
   - 阶段 3+: 每个用户故事一个阶段（按 PRD 中的优先级顺序）
   - 每个阶段包括：故事目标、独立测试标准、测试（如要求）、实现任务
   - 最终阶段: 收尾与优化
   - 所有任务必须遵循严格的检查清单格式（见下方任务生成规则）
   - 每个任务的清晰文件路径
   - 显示故事完成顺序的依赖章节
   - 每个故事的并行执行示例
   - 实施策略章节（MVP 优先、增量交付）

5. **报告**: 输出生成的 FD 路径和摘要：
   - 总任务数
   - 每个用户故事的任务数
   - 识别的并行机会
   - 每个故事的独立测试标准
   - 建议的 MVP 范围（通常只是用户故事 1）
   - 格式验证：确认所有任务遵循检查清单格式（复选框、ID、标签、文件路径）

任务生成上下文: $ARGUMENTS

FD 应该可以立即执行 - 每个任务必须足够具体，以便 LLM 可以在没有额外上下文的情况下完成它。

## 任务生成规则

**关键**: 任务必须按用户故事组织，以便独立实现和测试。

**测试是可选的**: 仅在功能规格中明确要求或用户要求 TDD 方法时才生成测试任务。

### 检查清单格式（必需）

每个任务必须严格遵循此格式：

```text
- [ ] [任务ID] [P?] [故事?] 描述及文件路径
```

**格式组件**:

1. **复选框**: 始终以 `- [ ]` 开头（Markdown 复选框）
2. **任务 ID**: 按执行顺序的序号（T001、T002、T003...）
3. **[P] 标记**: 仅在任务可并行时包含（不同文件，不依赖未完成的任务）
4. **[故事] 标签**: 仅用户故事阶段任务需要
   - 格式: [US1]、[US2]、[US3] 等（映射到 PRD 中的用户故事）
   - 设置阶段: 无故事标签
   - 基础架构阶段: 无故事标签
   - 用户故事阶段: 必须有故事标签
   - 收尾阶段: 无故事标签
5. **描述**: 带有确切文件路径的清晰操作

**示例**:

- ✅ 正确: `- [ ] T001 按实施计划创建项目结构`
- ✅ 正确: `- [ ] T005 [P] 实现认证中间件 src/middleware/auth.py`
- ✅ 正确: `- [ ] T012 [P] [US1] 创建 User 模型 domain/model/User.kt`
- ✅ 正确: `- [ ] T014 [US1] 实现 UserService domain/usecase/UserUseCase.kt`
- ❌ 错误: `- [ ] 创建 User 模型`（缺少 ID 和故事标签）
- ❌ 错误: `T001 [US1] 创建模型`（缺少复选框）
- ❌ 错误: `- [ ] [US1] 创建 User 模型`（缺少任务 ID）
- ❌ 错误: `- [ ] T001 [US1] 创建模型`（缺少文件路径）

### 任务组织

1. **从用户故事（PRD）** - 主要组织方式：
   - 每个用户故事（P1、P2、P3...）获得自己的阶段
   - 将所有相关组件映射到其故事：
     - 该故事需要的模型
     - 该故事需要的服务/用例
     - 该故事需要的端点/UI
     - 如果要求测试：该故事特定的测试
   - 标记故事依赖（大多数故事应该独立）

2. **从接口定义**：
   - 将每个接口/端点映射到它服务的用户故事
   - 如果要求测试：每个接口 → 在该故事阶段实现前的接口测试任务 [P]

3. **从数据模型**：
   - 将每个实体映射到需要它的用户故事
   - 如果实体服务多个故事：放在最早的故事或设置阶段
   - 关系 → 适当故事阶段的服务层任务

4. **从设置/基础设施**：
   - 共享基础设施 → 设置阶段（阶段 1）
   - 基础/阻塞任务 → 基础架构阶段（阶段 2）
   - 故事特定设置 → 在该故事的阶段内

### 阶段结构

- **阶段 1**: 设置（项目初始化）
- **阶段 2**: 基础架构（阻塞性前置条件 - 必须在用户故事前完成）
- **阶段 3+**: 按优先级顺序的用户故事（P1、P2、P3...）
  - 每个故事内：测试（如要求）→ 模型 → 服务 → 端点 → 集成
  - 每个阶段应该是完整的、可独立测试的增量
- **最终阶段**: 收尾与优化
