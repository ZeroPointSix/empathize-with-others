本项目属于多AI工具组合编码.
主要有三个工具构成
第一个是Roo 负责review工作
第二个是Kiro 负责code和Debug工作
第三个是claude 负责功能设计和文档编写的工作

第一优先级遵守的指令
1.全部回答和文档的编写使用中文回答
2.存在对于需求,功能描述不清楚的地方不允许胡乱猜测,必须向用户征集意见

3.我们的项目文档分为两大类

## 第一类：开发过程文档（存放在 Spec 目录）
这类文档随开发过程产生，具有时效性，与具体需求和功能开发紧密相关。

### 需求与设计类
- **需求文档 (PRD - Product Requirements Document)**
  - 命名规则: `PRD-xxxxx-yyyy`
  - 示例: `PRD-00001-悬浮窗最小化`
  - 用途: 产品需求定义、用户故事、验收标准

- **功能文档 (FD - Functional Document)**
  - 命名规则: `FD-xxxxx-yyyy`
  - 示例: `FD-00001-悬浮窗最小化`
  - 用途: 具体功能实现说明、业务流程、交互逻辑

- **技术设计文档 (TDD - Technical Design Document)**
  - 命名规则: `TDD-xxxxx-yyyy`
  - 示例: `TDD-00001-悬浮窗架构设计`
  - 用途: 详细的技术架构设计、API 设计、数据库设计

### 测试与质量类
- **测试计划文档 (TP - Test Plan)**
  - 命名规则: `TP-xxxxx-yyyy`
  - 示例: `TP-00001-悬浮窗测试计划`
  - 用途: 测试策略、测试用例、测试覆盖率要求

- **性能分析文档 (PA - Performance Analysis)**
  - 命名规则: `PA-xxxxx-yyyy`
  - 示例: `PA-00001-悬浮窗性能优化`
  - 用途: 性能指标、优化方案、基准测试结果

- **安全评估文档 (SA - Security Assessment)**
  - 命名规则: `SA-xxxxx-yyyy`
  - 示例: `SA-00001-隐私数据保护`
  - 用途: 安全风险评估、加密方案、权限设计

### 变更与部署类
- **变更记录 (CR - Change Record)**
  - 命名规则: `CR-xxxxx-yyyy`
  - 示例: `CR-00001-悬浮窗交互优化`
  - 用途: 记录需求变更、设计变更、影响分析

- **部署文档 (DD - Deployment Document)**
  - 命名规则: `DD-xxxxx-yyyy`
  - 示例: `DD-00001-生产环境部署`
  - 用途: 部署流程、环境配置、发布检查清单

### 总结类
- **回顾总结文档 (RR - Retrospective Report)**
  - 命名规则: `RR-xxxxx-yyyy`
  - 示例: `RR-00001-悬浮窗开发总结`
  - 用途: 项目复盘、经验教训、改进建议

---

## 第二类：项目长期文档（存放在 docs 目录）
这类文档是项目的长期资产，记录整体规范、架构设计和面向不同角色的指南。

### 架构与规范类
- **规范文档 (SD - Specification Document)**
  - 命名规则: `SD-xxxxx-yyyy`
  - 示例: `SD-00001-代码规范`
  - 用途: 技术规范、编码标准、接口约定、项目整体规范

- **API 文档 (API - API Documentation)**
  - 命名规则: `API-xxxxx-yyyy`
  - 示例: `API-00001-AI分析接口`
  - 用途: 接口定义、请求/响应格式、错误码说明

### 面向用户类
- **用户手册 (UM - User Manual)**
  - 命名规则: `UM-xxxxx-yyyy`
  - 示例: `UM-00001-悬浮窗使用指南`
  - 用途: 面向最终用户的功能使用说明

- **故障排查文档 (TSG - Troubleshooting Guide)**
  - 命名规则: `TSG-xxxxx-yyyy`
  - 示例: `TSG-00001-悬浮窗常见问题`
  - 用途: 常见问题、错误处理、调试指南

---

### 文档编号说明
- `xxxxx`: 5位数字编号，从00001开始递增
- `yyyy`: 简短的功能或需求描述，使用中文
- 同一需求的不同类型文档使用相同的编号
- 开发过程文档的编号与具体需求绑定
- 项目长期文档的编号按主题独立编号
