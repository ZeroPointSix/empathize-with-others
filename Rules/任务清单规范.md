## 1. 文档定位

任务清单（Task List）是**功能文档（FD）的核心组成部分**，用于将需求文档（PRD）和技术设计（TDD）转化为可执行的、依赖有序的任务列表。

### 1.1 任务清单的作用

任务清单是"需求的执行计划"，它：
- ✅ 将用户故事分解为具体的开发任务
- ✅ 明确任务之间的依赖关系和执行顺序
- ✅ 标识可并行执行的任务，提高开发效率
- ✅ 提供清晰的检查点，便于验证和测试
- ✅ 支持增量交付和MVP优先策略

### 1.2 与其他文档的关系

```
PRD (需求文档)
    ↓ 定义用户故事和优先级
TDD (技术设计)
    ↓ 定义技术架构和实现方案
FD (功能文档) = 任务清单
    ↓ 分解为可执行任务
实际开发
```

## 2. 文档结构

### 2.1 基本信息

每个任务清单必须包含以下基本信息：

```markdown
# 功能文档 (FD): [功能名称]

**文档编号**: FD-[xxxxx]-[功能名称]  
**创建日期**: [日期]  
**需求文档**: `文档/开发文档/PRD/PRD-[xxxxx]-[功能名称].md`  
**技术设计**: `文档/开发文档/TDD/TDD-[xxxxx]-[功能名称].md`
**状态**: [待开始/进行中/已完成]
**负责人**: [开发者名称]
```

### 2.2 任务格式说明

```markdown
## 任务格式说明

- **[P]**: 可并行执行（不同文件，无依赖）
- **[US?]**: 所属用户故事（例如 US1, US2, US3）
- 描述中包含确切的文件路径
```

### 2.3 路径约定

```markdown
## 路径约定

基于项目结构 `com.empathy.ai/`:
- **领域层**: `domain/model/`, `domain/repository/`, `domain/usecase/`
- **数据层**: `data/local/`, `data/repository/`
- **表现层**: `presentation/ui/screen/`, `presentation/viewmodel/`
```

## 3. 任务组织结构

### 3.1 阶段划分

任务清单必须按以下阶段组织：

#### 阶段 1: 基础设置

**目标**: 项目初始化和基础结构

**格式示例（扁平化）**：

```markdown
## 阶段 1: 基础设置

**目标**: 项目初始化和基础结构

- [ ] T001 创建功能相关的包结构
- [ ] T002 [P] 添加必要的依赖到 `build.gradle.kts`
- [ ] T003 [P] 创建 Hilt 模块配置 `di/MemoryModule.kt`

**检查点**: 基础包结构和依赖配置完成
```

**格式示例（层级化）**：

```markdown
## Phase 1: 基础设置

**目标**: 项目初始化和基础结构

- [ ] 1. 创建项目基础结构
  - 创建记忆系统相关的包结构
  - 添加Paging 3依赖
  - 创建MemoryModule Hilt配置
  - _需求: PRD-00003 基础架构_

- [ ] 1.1 创建包结构
  - 在 `domain/` 下创建 `model/memory/` 包
  - 在 `data/local/` 下创建 `entity/memory/` 包
  - 在 `data/local/dao/` 下创建记忆相关DAO
  
- [ ] 1.2 添加依赖
  - 在 `app/build.gradle.kts` 添加 Paging 3
  - 版本号使用 `libs.versions.toml` 管理
  
- [ ] 1.3 创建Hilt模块
  - 创建 `di/MemoryModule.kt`
  - 添加 @Module 和 @InstallIn 注解

**检查点**: 基础包结构和依赖配置完成
```

#### 阶段 2: 基础架构

**目标**: 必须在任何用户故事开始前完成的核心基础设施

```markdown
## 阶段 2: 基础架构（阻塞性前置条件）

**目标**: 必须在任何用户故事开始前完成的核心基础设施

**⚠️ 关键**: 此阶段完成前不能开始任何用户故事

- [ ] T004 创建领域模型 `domain/model/[Entity].kt`
- [ ] T005 [P] 创建数据库实体 `data/local/entity/[Entity]Entity.kt`
- [ ] T006 [P] 创建 DAO 接口 `data/local/dao/[Entity]Dao.kt`
- [ ] T007 创建 Repository 接口 `domain/repository/[Entity]Repository.kt`
- [ ] T008 实现 Repository `data/repository/[Entity]RepositoryImpl.kt`
- [ ] T009 在 DatabaseModule 中注册 DAO
- [ ] T010 在 RepositoryModule 中注册 Repository

**检查点**: 基础架构就绪 - 可以开始用户故事实现
```

#### 阶段 3+: 用户故事

**目标**: 按优先级实现用户故事

```markdown
## 阶段 3: 用户故事 1 - [标题] (优先级: P1) 🎯 MVP

**目标**: [简要描述此故事交付的内容]

**独立测试**: [如何验证此故事独立工作]

### 用户故事 1 的测试（可选 - 仅在要求测试时）⚠️

> **注意: 先编写测试，确保测试失败后再实现**

- [ ] T011 [P] [US1] 单元测试 `test/.../[UseCase]Test.kt`
- [ ] T012 [P] [US1] ViewModel 测试 `test/.../[ViewModel]Test.kt`

### 用户故事 1 的实现

- [ ] T013 [US1] 创建用例 `domain/usecase/[UseCase].kt`
- [ ] T014 [US1] 创建 UiState `presentation/ui/screen/[功能]/[功能]UiState.kt`
- [ ] T015 [US1] 创建 UiEvent `presentation/ui/screen/[功能]/[功能]UiEvent.kt`
- [ ] T016 [US1] 创建 ViewModel `presentation/viewmodel/[功能]ViewModel.kt`
- [ ] T017 [US1] 创建 Screen `presentation/ui/screen/[功能]/[功能]Screen.kt`
- [ ] T018 [US1] 添加导航路由到 NavGraph.kt

**检查点**: 用户故事 1 应该完全可用且可独立测试
```

#### 阶段 N: 收尾与优化

```markdown
## 阶段 N: 收尾与优化

**目标**: 影响多个用户故事的改进

- [ ] TXXX [P] 更新相关文档
- [ ] TXXX 代码清理和重构
- [ ] TXXX 性能优化
- [ ] TXXX 安全加固
```

### 3.2 依赖关系与执行顺序

```markdown
## 依赖关系与执行顺序

### 阶段依赖

- **阶段 1 (设置)**: 无依赖 - 可立即开始
- **阶段 2 (基础架构)**: 依赖设置完成 - 阻塞所有用户故事
- **阶段 3+ (用户故事)**: 都依赖基础架构完成
  - 用户故事可以并行进行（如果有人力）
  - 或按优先级顺序执行（P1 → P2 → P3）
- **阶段 N (收尾)**: 依赖所有期望的用户故事完成

### 用户故事依赖

- **用户故事 1 (P1)**: 基础架构完成后可开始 - 不依赖其他故事
- **用户故事 2 (P2)**: 基础架构完成后可开始 - 可能与 US1 集成但应独立可测
- **用户故事 3 (P3)**: 基础架构完成后可开始 - 可能与 US1/US2 集成但应独立可测

### 并行机会

- 所有标记 [P] 的任务可以并行执行
- 基础架构完成后，所有用户故事可以并行开始
- 不同用户故事可以由不同开发者并行处理
```

### 3.3 实施策略

```markdown
## 实施策略

### MVP 优先（仅用户故事 1）

1. 完成阶段 1: 设置
2. 完成阶段 2: 基础架构（关键 - 阻塞所有故事）
3. 完成阶段 3: 用户故事 1
4. **停止并验证**: 独立测试用户故事 1
5. 如果就绪则部署/演示

### 增量交付

1. 完成设置 + 基础架构 → 基础就绪
2. 添加用户故事 1 → 独立测试 → 部署/演示 (MVP!)
3. 添加用户故事 2 → 独立测试 → 部署/演示
4. 添加用户故事 3 → 独立测试 → 部署/演示
5. 每个故事增加价值而不破坏之前的故事
```

## 4. 任务编写规范

### 4.1 任务格式类型

任务清单支持两种格式：**扁平化格式**和**层级化格式**。根据任务复杂度选择合适的格式。

#### 4.1.1 扁平化格式（简单任务）

适用于：简单、明确、单步骤的任务

**格式**：
```text
- [ ] [任务ID] [P?] [故事?] 描述及文件路径
```

**格式组件**:

1. **复选框**: 始终以 `- [ ]` 开头（Markdown 复选框）
2. **任务 ID**: 按执行顺序的序号（T001、T002、T003...）
3. **[P] 标记**: 仅在任务可并行时包含（不同文件，不依赖未完成的任务）
4. **[故事] 标签**: 仅用户故事阶段任务需要
   - 格式: [US1]、[US2]、[US3] 等（映射到 PRD 中的用户故事）
   - 设置阶段: 无故事标签
   - 基础架构阶段: 无故事标签
   - 用户故事阶段: 必须有故事标签
   - 收尾阶段: 无故事标签
5. **描述**: 带有确切文件路径的清晰操作

**✅ 正确示例**:

```markdown
- [ ] T001 按实施计划创建项目结构
- [ ] T005 [P] 实现认证中间件 `src/middleware/auth.py`
- [ ] T012 [P] [US1] 创建 User 模型 `domain/model/User.kt`
- [ ] T014 [US1] 实现 UserService `domain/usecase/UserUseCase.kt`
```

**❌ 错误示例**:

```markdown
- [ ] 创建 User 模型                    # 缺少 ID 和故事标签
T001 [US1] 创建模型                     # 缺少复选框
- [ ] [US1] 创建 User 模型              # 缺少任务 ID
- [ ] T001 [US1] 创建模型               # 缺少文件路径
```

#### 4.1.2 层级化格式（复杂任务）

适用于：复杂、多步骤、需要详细说明的任务

**格式**：
```markdown
- [ ] [任务ID] [P?] [故事?] 父任务描述
  - 子任务描述1（详细步骤）
  - 子任务描述2（详细步骤）
  - 子任务描述3（详细步骤）
  - _需求: [需求追溯]_

- [ ] [任务ID.1] [P?] [故事?] 子任务1标题
  - 在 `具体文件路径` 中实现xxx
  - 添加xxx功能
  - 配置xxx参数
  - _需求: [需求追溯]_
  
- [ ] [任务ID.2] [P?] [故事?] 子任务2标题
  - 实施步骤1
  - 实施步骤2
  - _需求: [需求追溯]_
```

**层级化格式组件**:

1. **父任务**: 概括性描述，定义任务组的目标
2. **子任务列表**: 父任务下的详细步骤（使用缩进的bullet points）
3. **子任务展开**: 每个子任务可以展开为独立的任务项
4. **需求追溯**: 使用 `_需求: xxx_` 标记关联的需求或设计文档
5. **任务编号**: 父任务使用整数（1, 2, 3），子任务使用小数（1.1, 1.2, 1.3）

**✅ 层级化格式示例**:

```markdown
## Phase 1: 基础增强

- [ ] 1. 创建配置文件和默认值
  - 创建字段映射配置文件
  - 创建配置加载逻辑
  - 定义默认值常量
  - _需求: PRD-00003 2.5, 6.1, 6.2, 8.2_

- [ ] 1.1 创建字段映射配置文件
  - 在 `app/src/main/assets/` 目录创建 `field_mappings.json`
  - 定义 AnalysisResult 字段映射（replySuggestion、strategyAnalysis、riskLevel）
  - 定义 SafetyCheckResult 字段映射（isSafe、triggeredRisks、suggestion）
  - 定义 ExtractedData 字段映射（facts、redTags、greenTags）
  - _需求: PRD-00003 2.5, 8.2_

- [ ] 1.2 创建 FieldMappingConfig 对象
  - 在 `AiRepositoryImpl.kt` 中创建 FieldMappingConfig 单例对象
  - 实现 load() 方法从 assets 加载配置文件
  - 实现配置缓存逻辑（启动时加载一次）
  - 实现 getDefaultMappings() 方法提供硬编码的默认配置
  - 添加错误处理和日志记录
  - _需求: PRD-00003 2.5, 8.2_

- [ ] 2. 优化现有解析方法
  - 增强 preprocessJsonResponse 方法
  - 优化 mapChineseFieldNames 方法
  - _需求: PRD-00003 1.1, 1.2, 1.3, 1.4, 2.1, 2.2, 2.3, 2.4_

- [ ] 2.1 增强 preprocessJsonResponse 方法
  - 简化正则表达式，提高性能
  - 优化字符串操作（使用 StringBuilder）
  - 添加最终验证步骤 validateAndFixJson()
  - 优化日志输出（限制长度，条件输出）
  - 保持现有的清洗逻辑（移除 Markdown、提取 JSON、修复格式错误）
  - _需求: PRD-00003 1.1, 1.2, 1.3, 1.4_
```

#### 4.1.3 格式选择指南

**使用扁平化格式**：
- ✅ 任务步骤单一明确
- ✅ 文件路径清晰
- ✅ 不需要详细说明
- ✅ 示例：创建实体类、添加依赖、注册模块

**使用层级化格式**：
- ✅ 任务包含多个步骤
- ✅ 需要详细的实施说明
- ✅ 需要追溯需求文档
- ✅ 任务之间有逻辑分组
- ✅ 示例：复杂的算法实现、多文件协同修改、系统集成

**混合使用**：
- 在同一个任务清单中，可以根据任务复杂度混合使用两种格式
- 基础设置阶段：通常使用扁平化格式
- 用户故事实现阶段：根据复杂度选择格式
- 性能优化阶段：通常使用层级化格式

### 4.2 任务描述要求

每个任务描述必须：

1. **明确具体**: 清楚说明要做什么
2. **包含路径**: 指明具体的文件路径（使用反引号包裹）
3. **可独立执行**: 不依赖未说明的上下文
4. **可验证**: 有明确的完成标准

**示例**:

```markdown
✅ 好的描述（扁平化）:
- [ ] T013 [US1] 创建用例 `domain/usecase/AnalyzeChatUseCase.kt`，实现聊天分析逻辑

✅ 好的描述（层级化）:
- [ ] 1.3 创建 DefaultValues 对象
  - 在 `AiRepositoryImpl.kt` 中创建 DefaultValues 单例对象
  - 定义 ANALYSIS_RESULT 默认值
  - 定义 SAFETY_CHECK_RESULT 默认值
  - 定义 EXTRACTED_DATA 默认值
  - _需求: PRD-00003 6.1, 6.2_

❌ 不好的描述:
- [ ] T013 [US1] 创建用例
- [ ] T013 [US1] 实现分析功能
- [ ] 1.3 创建对象（缺少文件路径和详细步骤）
```

### 4.3 需求追溯

层级化格式中，使用 `_需求: xxx_` 标记关联的需求或设计文档：

**格式**：
```markdown
- _需求: PRD-xxxxx 章节号_
- _需求: TDD-xxxxx 章节号_
- _需求: 功能点描述_
```

**示例**：
```markdown
- _需求: PRD-00003 2.5, 8.2_
- _需求: TDD-00003 数据层设计_
- _需求: 字段映射配置、默认值支持_
```

**作用**：
- ✅ 追溯任务到需求文档
- ✅ 验证任务完整性
- ✅ 便于需求变更影响分析
- ✅ 支持测试用例设计

## 5. 任务生成规则

### 5.1 从用户故事生成任务

**主要组织方式**：每个用户故事（P1、P2、P3...）获得自己的阶段

**映射规则**：
- 该故事需要的模型 → 模型创建任务
- 该故事需要的服务/用例 → 用例创建任务
- 该故事需要的端点/UI → UI创建任务
- 如果要求测试 → 该故事特定的测试任务

### 5.2 从技术设计生成任务

**从接口定义**：
- 将每个接口/端点映射到它服务的用户故事
- 如果要求测试：每个接口 → 在该故事阶段实现前的接口测试任务 [P]

**从数据模型**：
- 将每个实体映射到需要它的用户故事
- 如果实体服务多个故事：放在最早的故事或设置阶段
- 关系 → 适当故事阶段的服务层任务

**从设置/基础设施**：
- 共享基础设施 → 设置阶段（阶段 1）
- 基础/阻塞任务 → 基础架构阶段（阶段 2）
- 故事特定设置 → 在该故事的阶段内

### 5.3 测试任务生成规则

**重要**: 测试是可选的，仅在功能规格中明确要求或用户要求 TDD 方法时才生成测试任务。

**测试任务位置**：
- 单元测试：在实现任务之前
- 集成测试：在相关模块实现之后
- UI测试：在UI实现之后

**测试任务格式**：
```markdown
- [ ] T011 [P] [US1] 单元测试 `test/.../[UseCase]Test.kt`
- [ ] T012 [P] [US1] ViewModel 测试 `test/.../[ViewModel]Test.kt`
```

## 6. 文档存放与命名

### 6.1 存放路径

```
文档/
└── 开发文档/
    └── FD/
        └── FD-xxxxx-功能名称.md
```

### 6.2 命名规则

- **格式**: `FD-xxxxx-功能名称.md`
- **编号**: 5位数字，从00001开始递增
- **功能名称**: 简短的中文描述
- **编号一致性**: 与对应的PRD、TDD使用相同编号

**示例**:
```
FD-00001-悬浮窗功能.md
FD-00002-设置功能.md
FD-00003-联系人画像记忆系统.md
```

## 7. 文档生成流程

### 7.1 准备阶段

1. **查找对应文档**：
   - 需求文档（PRD）：获取用户故事和优先级
   - 技术设计（TDD）：获取技术架构和实现方案

2. **确定文档编号**：
   - 使用与PRD、TDD相同的编号
   - 创建FD文档路径

### 7.2 生成阶段

1. **加载设计文档**：
   - **必需**: TDD（技术栈、库、结构）、PRD（用户故事及优先级）
   - **可选**: data-model.md（实体）、contracts/（API 端点）

2. **执行任务生成**：
   - 加载 TDD 并提取技术栈、库、项目结构
   - 加载 PRD 并提取用户故事及其优先级（P1、P2、P3 等）
   - 如果 data-model.md 存在：提取实体并映射到用户故事
   - 如果 contracts/ 存在：将端点映射到用户故事
   - 生成按用户故事组织的任务
   - 生成显示用户故事完成顺序的依赖图
   - 为每个用户故事创建并行执行示例
   - 验证任务完整性

3. **生成文档**：
   - 使用模板结构
   - 填写所有必需章节
   - 确保所有任务遵循检查清单格式
   - 每个任务包含清晰文件路径

### 7.3 验证阶段

1. **格式验证**：
   - 确认所有任务遵循检查清单格式（复选框、ID、标签、文件路径）
   - 确认任务ID连续且唯一
   - 确认用户故事标签正确

2. **完整性验证**：
   - 每个用户故事有所有需要的任务
   - 每个用户故事可独立测试
   - 依赖关系清晰明确

3. **输出报告**：
   - 总任务数
   - 每个用户故事的任务数
   - 识别的并行机会
   - 每个故事的独立测试标准
   - 建议的 MVP 范围

## 9. 注意事项

### 9.1 任务编写注意事项

- ✅ [P] 任务 = 不同文件，无依赖
- ✅ [US?] 标签将任务映射到特定用户故事以便追踪
- ✅ 每个用户故事应该独立可完成和可测试
- ✅ 验证测试失败后再实现
- ✅ 每个任务或逻辑组完成后提交
- ✅ 在任何检查点停止以独立验证故事

### 9.2 避免的问题

- ❌ 模糊任务：缺少具体文件路径
- ❌ 同文件冲突：多个任务修改同一文件
- ❌ 破坏独立性：跨故事依赖导致无法独立测试
- ❌ 缺少检查点：无法及时验证质量
- ❌ 任务粒度过大：难以跟踪进度

### 9.3 质量保证

- 每个任务必须可独立执行
- 每个任务必须有明确的完成标准
- 每个用户故事必须可独立测试
- 每个阶段必须有明确的检查点
- 整体任务清单必须支持增量交付

## 11. 最佳实践

### 11.1 何时使用层级化格式

**推荐场景**：

1. **复杂的技术实现**
   - 多步骤的算法实现
   - 需要详细配置的功能
   - 涉及多个文件协同修改

2. **系统集成任务**
   - AI响应解析增强
   - 数据库迁移
   - 第三方服务集成

3. **性能优化任务**
   - 需要分析、优化、验证多个步骤
   - 需要详细记录优化策略

4. **需要详细追溯的任务**
   - 需要关联多个需求点
   - 需要详细的实施说明
   - 需要明确的验收标准

**示例对比**：

```markdown
❌ 扁平化格式（不够清晰）:
- [ ] T050 [US2] 实现summarizeForContact方法

✅ 层级化格式（清晰明确）:
- [ ] 2.1 实现summarizeForContact方法
  - 在 `SummarizeDailyConversationsUseCase.kt` 中实现
  - 查询指定联系人的未总结对话记录
  - 构建总结Prompt（包含对话记录和联系人画像）
  - 调用AI服务获取总结
  - 解析AI响应并更新联系人数据
  - 保存总结记录到数据库
  - 添加错误处理和降级方案
  - _需求: PRD-00003 US2 每日自动总结_
```

### 11.2 任务粒度控制

**扁平化格式**：
- 每个任务应该是一个独立的文件操作
- 任务时间：15-60分钟
- 任务复杂度：低到中等

**层级化格式**：
- 父任务：概括性目标，1-4小时
- 子任务：具体步骤，15-60分钟
- 任务层级：最多3层（父任务 → 子任务 → 实施步骤）

### 11.3 检查点设置

每个阶段必须有明确的检查点：

```markdown
**检查点**: [验证标准]
- ✅ [功能验收标准1]
- ✅ [功能验收标准2]
- ✅ [技术验收标准1]
```

**示例**：

```markdown
**检查点**: 用户故事 1 应该完全可用且可独立测试
- ✅ 对话记录能自动保存
- ✅ 保存失败不影响AI分析
- ✅ 可以查询历史对话记录
```

### 11.4 并行任务识别

**标记规则**：
- 不同文件 + 无依赖 = [P]
- 同一文件 = 不能并行
- 有依赖关系 = 不能并行

**示例**：

```markdown
✅ 可以并行（不同文件，无依赖）:
- [ ] T014 [P] 创建ConversationLogEntity `data/local/entity/ConversationLogEntity.kt`
- [ ] T015 [P] 创建DailySummaryEntity `data/local/entity/DailySummaryEntity.kt`
- [ ] T016 [P] 创建FailedSummaryTaskEntity `data/local/entity/FailedSummaryTaskEntity.kt`

❌ 不能并行（同一文件）:
- [ ] T021 创建数据库Migration 3→4脚本 `data/local/DatabaseModule.kt`
- [ ] T022 创建数据库Migration 4→5脚本 `data/local/DatabaseModule.kt`

❌ 不能并行（有依赖）:
- [ ] T024 创建ConversationRepository接口 `domain/repository/ConversationRepository.kt`
- [ ] T027 实现ConversationRepositoryImpl `data/repository/ConversationRepositoryImpl.kt`
```
