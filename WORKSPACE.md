#### TD-001: Room数据库迁移策略需要完善

**背景**：
- 当前使用 `fallbackToDestructiveMigration()` 允许破坏性迁移
- 这是MVP开发阶段的临时方案，会删除所有旧数据
- 正式发布后必须保护用户数据，不能丢失

**必须完成的工作**：

1. **移除破坏性迁移配置**
   ```kotlin
   // ❌ 当前（开发阶段）
   .fallbackToDestructiveMigration()
   
   // ✅ 正式发布
   // 移除此行，强制使用Migration脚本
   ```

2. **启用Schema导出**
   ```kotlin
   @Database(
       entities = [...],
       version = 5,
       exportSchema = true  // 改为 true
   )
   ```
   - 配置schema导出目录
   - 将schema JSON文件纳入版本控制
   - 用于版本管理和测试验证

3. **编写完整的Migration测试**
   ```kotlin
   @Test
   fun migrate3To4_containsCorrectData() {
       // 使用 Room Migration Testing 库
       // 测试从版本3升级到版本4的数据完整性
   }
   ```

4. **验证所有升级路径**
   - 测试 1→2, 2→3, 3→4, 4→5 的每个迁移
   - 测试跨版本升级（如 1→5）
   - 验证数据不丢失、外键约束正确

**相关文件**：
- `app/src/main/java/com/empathy/ai/di/DatabaseModule.kt`
- `app/src/main/java/com/empathy/ai/data/local/AppDatabase.kt`

- [IMPL-00005-Room数据库迁移错误修复.md](../文档/开发文档/IMPL/IMPL-00005-Room数据库迁移错误修复.md)
