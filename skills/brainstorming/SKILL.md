# 头脑风暴

## 激活时机

当满足以下条件时自动激活此技能：
- 需要设计新功能
- 遇到技术难题需要多角度思考
- 需要探索多种实现方案
- 需要评估技术选型
- 需要拆解复杂问题

## 头脑风暴流程

### 1. 明确问题

```
问题陈述：
- 我们要解决什么问题？
- 当前状况是什么？
- 期望结果是什么？
- 有什么约束条件？
```

### 2. 自由发散

```
发散规则：
- 追求数量，暂不评判
- 欢迎疯狂的想法
- 建立在他人想法之上
- 不批评、不限制
```

### 3. 分类整理

```
分类维度：
- 技术可行性
- 实现成本
- 风险评估
- 优先级排序
```

### 4. 收敛决策

```
决策标准：
- 技术可行性
- 资源可用性
- 时间/成本约束
- ROI 评估
```

## 问题拆解技巧

### MECE 原则

```
MECE: Mutually Exclusive, Collectively Exhaustive
相互独立，完全穷尽

示例：如何提升用户体验？

拆解为：
1. 性能优化
   - 加载速度
   - 响应时间
   - 动画流畅度

2. 功能完善
   - 核心功能
   - 辅助功能
   - 高级功能

3. 交互设计
   - 操作便捷性
   - 视觉反馈
   - 错误处理

4. 视觉设计
   - 布局美观
   - 色彩搭配
   - 动效设计
```

### 5W2H 分析法

```
What:   是什么问题？
Why:    为什么会出现？
Who:    涉及哪些用户？
When:   何时发生？
Where:  在哪里发生？
How:    如何解决？
How much: 成本多少？
```

### 第一性原理

```
从基本事实出发，层层推导

示例：设计一个文件上传功能

第一性原理分析：
1. 文件的本质是什么？
   - 二进制数据

2. 上传的本质是什么？
   - 数据从客户端传输到服务器

3. 需要考虑什么？
   - 数据完整性（校验）
   - 传输效率（分片、压缩）
   - 存储位置（本地/云）
   - 安全性（病毒扫描、权限）

4. 用户体验？
   - 进度显示
   - 断点续传
   - 错误处理
```

## 方案探索方法

### SCAMPER 法

```
S - Substitute 替代
  - 能否替换技术栈？
  - 能否使用现有库？

C - Combine 组合
  - 能否组合多个方案？
  - 能否集成现有服务？

A - Adapt 调整
  - 其他项目如何解决？
  - 能否借鉴其他领域？

M - Modify 修改
  - 修改哪个参数？
  - 改变什么流程？

P - Put to other use 其他用途
  - 这个技术还能用在哪？
  - 能否复用现有组件？

E - Eliminate 消除
  - 删减不必要功能？
  - 简化流程？

R - Reverse 反转
  - 反过来思考？
  - 重新排序？
```

### 六顶思考帽

```
⚪ 白帽：客观事实
  - 现有技术能力
  - 资源约束
  - 数据支持

🔴 红帽：情感直觉
  - 用户体验
  - 团队感受
  - 市场反应

⚫ 黑帽：谨慎风险
  - 潜在问题
  - 技术风险
  - 资源风险

🟡 黄帽：积极乐观
  - 方案优势
  - 成功案例
  - 价值收益

🟢 绿帽：创意创新
  - 新思路
  - 替代方案
  - 突破性想法

🔵 蓝帽：控制流程
  - 思考过程
  - 决策流程
  - 总结梳理
```

## 技术选型分析

### 决策矩阵

```
评分维度：
- 功能完整性
- 学习曲线
- 社区活跃度
- 性能表现
- 生态成熟度
- 团队经验

示例：状态管理方案选择

           | Redux | Zustand | Context+useState |
-----------|-------|---------|-----------------|
功能完整性  | 9/10  | 7/10    | 5/10           |
学习曲线    | 4/10  | 9/10    | 8/10           |
社区活跃度  | 10/10 | 8/10    | 10/10          |
性能表现    | 7/10  | 10/10   | 6/10           |
团队经验    | 9/10  | 5/10    | 9/10           |
-----------|-------|---------|-----------------
总分        | 39/50 | 39/50   | 38/50          |

结论：根据项目特点选择
```

### 优缺点分析

```
方案A：微服务架构

优点：
+ 独立部署
+ 技术栈灵活
+ 故障隔离
+ 易于扩展

缺点：
- 分布式复杂
- 运维成本高
- 数据一致性挑战
- 调试困难

适用场景：
✅ 大型团队（>20人）
✅ 复杂业务
✅ 高并发需求
✅ 快速迭代

不适用场景：
❌ 小型项目
❌ 单一团队
❌ 初创公司
```

## 创意发散技巧

### 随机词联想法

```
1. 选择一个随机词
2. 强制关联当前问题
3. 产生新想法

示例：如何改进登录功能？

随机词：生物识别

联想想法：
- 指纹登录
- 面部识别
- 虹膜扫描
- 声纹验证
- 行为生物识别（打字节奏）
```

### 逆向思维

```
从相反方向思考

示例：如何提升系统性能？

正向思考：
- 优化算法
- 增加缓存
- 异步处理

逆向思考：
- 如何让系统更慢？
  - 添加不必要的循环
  - 同步阻塞操作
  - 不使用索引

→ 避免这些操作
```

### 类比迁移

```
从其他领域借鉴

示例：用户权限设计

类比：门禁系统
- 门禁卡 → API Token
- 权限级别 → 角色权限
- 访客登记 → 临时授权
- 审计日志 → 操作记录
```

## 最佳实践

### ✅ 应该做的

```
1. 创造安全的头脑风暴环境
   - 不批评、不评判
   - 欢迎疯狂想法
   - 鼓励建立他人想法

2. 追求数量而非质量
   - 先发散后收敛
   - 后期筛选评估

3. 多角度思考
   - 用户视角
   - 技术视角
   - 商业视角

4. 可视化思路
   - 使用白板/便利贴
   - 思维导图
   - 流程图
```

### ❌ 不应该做的

```
1. 过早评判
   - "这个不行"
   - "成本太高"
   - "技术实现不了"

2. 固守思维定式
   - "我们一直是这么做的"
   - "这是标准做法"

3. 忽略约束条件
   - 技术能力
   - 时间成本
   - 团队规模
```
